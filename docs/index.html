Final Project Website

Medical translation where it is needed most.

EDGE MD is designed for clinics and field settings where doctor-patient communication can break down due to language barriers. The goal is a portable English-Spanish translator that runs on an embedded edge pipeline.

Why this matters

From our README motivation: when providers and patients do not share a language, care quality and speed can drop. Relying on ad-hoc interpreters is not always safe or practical in urgent environments.

SRS and HRS Validation

Validated against quantified requirements with direct proof artifacts.

SRS validation (software)

  • SRS-01 (STT accuracy): tested two-sentence low-noise input against ground truth; observed complete transcription match in demo run. Outcome: pass for demonstrated sample.
  • SRS-02 (translation semantic quality): compared output to expected medical phrasing using BLEU-style checks and native-speaker review during demo. Outcome: pass for demonstrated test phrases.
  • Proof: demo video, audio output clip, and pipeline screenshots in this page's evidence section.

HRS validation (hardware)

  • HRS-02 (audio input path): microphone capture tested via STM32 + Pi logging and waveform inspection. Outcome: pass for low-noise speech capture.
  • HRS-03 (near real-time behavior): full capture-to-output cycle demonstrated in hardware-in-loop MVP testing. Outcome: pass for demo scenarios.
  • Proof: logic analyzer and oscilloscope captures, bench setup photos, and demo videos.

Conclusion and reflection

Justin

While satisfied with the progress we made, I would have laid out a much clearer plan from the beginning and focused more on testing between each stage. For many sessions we hit the same wall repeatedly before switching approaches. A stricter debug checklist and a running log of what worked (and why) would have reduced repeated dead-ends. Even with those challenges, I am proud of the progress and the amount we learned.

Tyrone

This project strengthened my understanding of how to split complex embedded systems into testable units and verify interfaces with lab instruments. For future builds, I would push for a clearer progress map and more explicit task tracking to reduce rework around tricky integration areas like UART/SPI communication. The team dynamic was a major strength, and I now have a much more practical feel for how these protocols should behave in real hardware.

Damodar (Dammi)

Our core vision was a handheld, fully edge-based translation device. We successfully demonstrated audio capture, translation, and output across the pipeline, but we could not fully unify every subsystem into the final seamless flow on the same timeline. We also faced repeated hardware issues (multiple faulty microphones and amplifiers), which consumed valuable debugging time. The project still delivered a strong learning experience in hardware/software co-design, system-level debugging, and planning for integration risk in future builds.
